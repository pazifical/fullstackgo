Weil einfach einfach einfach ist

Je nachdem, welche Programmiersprache du bereits gelernt hast, ist dein Empfinden von Einfachheit sicherlich vorgeprägt. Aber was ist im Kontext von Software-Engineering überhaupt mit Einfachheit und Komplexität gemeint? Und was hat das ganze überhaupt mit Go zu tun?


Einfachheit als Gegenstück zur Komplexität

Im Englischen ist es einfacher: Es gibt das Wort easy und das Wort simple. Im Deutschen verwenden wir irgendwie für beide Bedeutungen das Wort einfach... Happy und lucky lassen grüßen...

Wir wollen das Wort Einfachheit im Kontext der Software-Entwicklung also als Gegenstück zur Komplexität vertehen. Einfache Programmiersprachen sind leicht lesbar. Einfache Programmiersprachen haben wenig Keywords. Einfache Programmiersprachen können einfach zu lernen sein. Das heißt aber nicht, dass sie auch einfach zu meistern sind.


Go: Simplicity by Design

Die offizielle Webseite go.dev bewirbt Go wie folgt: Build simple, secure, scalable systems with Go. Die Einfachheit (simple) ist also ein Kern-Feature der Programmiersprache.

There should be one-- and preferably only one --obvious way to do it.
Wem kommt diese Aufforderung bekannt vor? Genau! Sie stammt aus dem allseits bekannten Zen Of Python. Wer auch nur ein wenig mit der Programmiersprache Python gearbeitet hat, wird sich aber stark wundern, wie viele Möglichkeiten es in Python gibt, das gleiche zu tun. Schauen wir uns das ganze mal in einem Beispiel an:

Aufgabe:
Filtere eine Liste aus Tieren so, dass eine neue Liste mit den Tieren, die mit "K" beginnen erzeugt wird. 

tiere = ["Kuh", "Katze", "Hund", "Pferd", "Krebs"]

# Variante 1: Der altmodische Klassiker
tiere_mit_k = []
for i in range(len(tiere)):
    tier = tiere[i]
    if tier.startswith("K"):
        tiere_mit_k.append(tier)

# Variante 2: Der modernere Ansatz
tiere_mit_k = []
for tier in tiere:
    if tier.startswith("K"):
        tiere_mit_k.append(tier)

# Variante 3: Der pythonische Ansatz  
tiere_mit_k = [tier for tier in tiere if tier.startswith("K")]

# Variante 4: Der funktionale Ansatz  
tiere_mit_k = list(filter(lambda x: x.startswith("K"), tiere))

Leider sind das noch nicht mal alle Möglichkeiten, wie man diese Aufgabe mit Python bewältigen kann... Auf den unerfahrenen Entwickler mag das noch wie ein Vorteil wirken. Immerhin kann man es genau so machen, wie man es gerade am liebsten mag. Aber sollte Quellcode wirklich diese persönliche Note haben?

Es muss nicht mal ein anderer Entwickler sein, der sich an einem bestimmten Stil stört. Möglicherweise findet der Entwickler, der den Code ursprünlich geschrieben hat, zu einem anderen Zeitpunkt einen ganz anderen Stil viel besser. Und plötzlich wird ein und das gleiche auf dutzend verschiedene Weisen geschrieben.

Python ist jedoch nicht die einzige Programmiersprache, bei der das so ist: Java, JavaScript, C#, C++, Rust und viele mehr fallen der gleichen Komplexität zum Opfer. Wer die Wahl hat, hat die Qual. Die Qual hat meistens die Person, die den Code dann verstehen muss.

Der Softwareentwickler Robert C. Martin (Uncle Bob) hat zwar diverse umstrittene und zum Teil nicht mehr ganz zeitgemäße Aussagen getroffen, aber eine Aussage steht weiter fest: Quellcode wird sehr viel häufiger gelesen als geschrieben.

Aber wie löst man die Aufgabe denn dann mit Go? Es gibt nur eine Variante: 

animals = []string{"Kuh", "Katze", "Hund", "Pferd", "Krebs"}

animalsWithK = make([]string, 0)
for _, animal := range animals {
  animalsWithK = append(animalsWithK, animal)
}

Ich geb zu: Elegant sieht das ganze nicht aus. Schreibt man einen der obigen Python-Einzeiler, fühlt man sich weitaus clever, nicht wahr?

Der große Vorteil bei Go ist jedoch, dass jeder Go-Entwickler diesen Code so schreibt. Das ist ein weiterer Grund dafür, dass praktisch jeder Go-Code gleich aussieht und man deshalb sehr schnell und einfach fremden Code lesen und verstehen kann.


Fazit

Beim Design der Programmiersprache Go und seiner Standardbibliothek wird penibel darauf geachtet, dass es tatsächlich nur eine Variante gibt, eine bestimmte Logik in Quellcode abzubilden.

Nach anfänglichem Zweifel und liebäugeln mit eleganteren Programmiersprachen wie Rust, bin ich inzwischen sehr froh, den Go-Weg eingeschlagen zu haben. Mein Code von vor zwei Jahren liest sich so, als hätte ich ihn gestern geschrieben. Weniger Komplexität in der Programmiersprache bedeutet weniger kognitive Last, sodass man sich stärker auf die Lösung des eigentlichen Problems konzentrieren kann.
